139// The reason we use jobs like this is to make sure we can tweak how much is
140// happening at any given time.
141//
142// On the Server level, we'll need to cooperate with all the databases in a
143// shared pool of workers. So, we need to come up with a design for the view
144// updaters to work within this limitation.
145//
146// Integrity scan is simple: Have a shared structure on Storage that keeps track
147// of all integrity scan results. It can check for an existing value and return,
148// or make you wait until the job is finished. For views, I suppose the best
149// that can be done is a similar approach, but the indexer's output is the last
150// transaction id it synced. When a request comes in, a check can be done if
151// there are any docs outdated, if so, the client can get the current transaction id
152// and ask the ViewScanning service manager to wait until that txid is scanned.
